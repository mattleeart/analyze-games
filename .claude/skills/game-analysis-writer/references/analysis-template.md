# 게임 분석서 작성 템플릿

기자 리뷰 수준의 장문 분석서를 위한 8개 섹션 구조다. 모든 섹션을 자세히 채운다. 한국어, 최소 8000자(목표 1.5만~2.5만자).

## 분량과 깊이 원칙
- 게임플레이 메카닉은 수치·시스템 이름·동작 메커니즘까지 정밀하게 서술한다("재미있다" 같은 막연한 표현 금지).
- 추측은 "[추정]"으로 명시한다. 확인된 사실만 단정한다.
- 평론 인용은 매체+점수(+가능하면 URL), 판매/매출/플레이어 통계는 출처(매체명+연도)를 병기한다.
- 깊이에 대한 판단이 필요하면 항상 더 디테일하고 정보량 많은 쪽을 택한다.

## 마크다운 표기
작품명·게임명·매체명은 《 》로 표기한다(꺾쇠 `<>` 금지 — HTML 태그로 깨짐). 부등호는 자연어("미만"/"이상")로 쓴다. 자세한 규칙은 SKILL.md 참조.

## 섹션 구조

### 1. 게임 개요
- 개발사/퍼블리셔, 출시일(플랫폼별), 장르
- 디렉터/리드/주요 크레딧
- 개발 기간, 규모(인력/예산), 엔진/기술 스택

### 2. 게임 설명 (이 게임이 무엇인가)
- 컨셉, 세계관, 줄거리(스포일러는 [스포일러] 표시)
- 캐릭터/주요 NPC
- 무드/톤/아트 디렉션
- 사운드/음악(작곡가, 대표곡)

### 3. 핵심 시스템 / 메카닉 (가장 자세하게)

**이 섹션은 반드시 "코어 게임플레이 루프"부터 시작한다.** 플레이어가 매 순간(대략 30초~몇 분 단위)에 실제로 *무엇을 반복하는지* 손에 잡히게 묘사하라 — 어떤 입력을 하고, 무엇을 판단하며, 어떤 즉각 보상을 받고, 그 짧은 반복이 어떻게 더 큰 진행(빌드 성장, 맵 정복, 서사 전개)으로 누적되는지. 이 루프 묘사가 빠지면 분석이 기능 나열식 "설명서"로 전락한다 — 게임이 *왜 손에서 놓기 어려운지*를 설명하는 것이 이 분석의 심장이다. 예: 로그라이트라면 "한 판(run)을 시작 → 방을 클리어하며 즉시 강화 선택 → 죽음 → 영구 성장 → 다시 시작"의 사이클과 각 단계의 의사결정·보상 타이밍을 구체적으로.

그 위에 다음을 쌓는다:
- 진행 구조(챕터/맵/오픈월드/로비/메타)
- 전투·주요 메카닉의 정밀한 동작(수치·시스템 이름·작동 방식)
- 자원 관리, 경제, 진척도 시스템
- 멀티/협력(해당 시), 라이브 운영/시즌패스/MTX(해당 시)
- 난이도·접근성 옵션, UI/UX 철학

### 4. 평가
- Metacritic 평론/유저 스코어, 주요 평론 인용(IGN, GameSpot, Eurogamer, PC Gamer, Polygon, Edge, Kotaku, RPS, OpenCritic 등)
- 수상 이력(TGA, BAFTA, GDCA, DICE, IGF 등)
- 상업 지표(판매량, 동접, 매출, 플레이어 수)
- 유저 평가(Steam, Reddit, 메가스레드), 평론-유저 괴리(있다면)

### 5. 성공 요인 분석 (핵심)
- 디자인 측면 성공 요인
- 마케팅/출시 전략, 타이밍/시장 환경
- 개발·운영 방법론
- 커뮤니티/IP 효과
- 동시기 경쟁작 대비 차별점

### 6. 비평적 시각 / 한계 / 논란
- 디자인적 약점, 운영/논란 이슈, 평가가 갈리는 지점

### 7. 영향과 유산
- 장르에 미친 영향, 후속작/모방작/장르 확장
- 산업적·문화적 의미

### 8. 부록
- GDC 강연/포스트모템 링크, 주요 인터뷰
- 핵심 통계 표, 참고 자료 목록(영어권 매체 중심)

## 리서치 효율 (quota 절약)
- `WebSearch`를 주력으로 8~12회. 검색 요약만으로 대부분 확보된다.
- `WebFetch`는 깊이가 꼭 필요한 핵심 3~5개로 제한. 전체 페이지를 가져와 토큰을 크게 쓴다.
- 같은 내용을 반복 검색하지 말고 키워드를 잘 구성한다.

## 검색 키워드 힌트
`"<게임명>" review`, `GDC postmortem`, `sales figures`, `director interview`, `design analysis`, `critical reception`, `awards`, `Reddit megathread`, `gameplay systems mechanics`
