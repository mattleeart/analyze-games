# Game Analysis Writer

성공한 게임을 "기자 리뷰 수준"의 장문 분석서로 정리하는 스킬이다. 단건으로 한 게임을 깊게 쓸 수도 있고, 워커+OS 스케줄러를 깔아 수십~수백 개를 사람 개입 없이 무한 자동 생성할 수도 있다.

작업 규모를 먼저 판단하라:
- **게임 1~3개** → 직접 작성 모드 (아래 "단건 작성")
- **게임 수십 개 이상 / "자동으로" / "멈추지 말고" / "밤새"** → 대량 자동화 모드 (`references/automation-setup.md`)

## 단건 작성

한 게임의 분석서를 직접 작성할 때의 절차다.

1. **리서치** — `WebSearch`를 주력으로 8~12회. 영어권 매체(IGN, GameSpot, Polygon, Eurogamer, PC Gamer, RPS, Edge, Kotaku, Game Informer, GamesIndustry.biz, Bloomberg, Variety, GDC Vault, Famitsu 영문) 위주. 깊이가 필요한 핵심 출처(GDC 포스트모템, 개발자 인터뷰, 판매 데이터)만 `WebFetch` 3~5회로 보강한다. WebFetch는 전체 페이지를 가져와 토큰을 크게 쓰므로 남용하지 마라 — 같은 내용을 반복 검색하지 말고 키워드를 잘 구성해 한 번에 충분히 얻는다.

2. **작성** — `references/analysis-template.md`의 8개 섹션을 모두 채운다. 한국어, 최소 8000자(목표 1.5만~2.5만자). 게임플레이 메카닉은 수치·시스템 이름·동작 메커니즘까지 정밀하게. 추측은 "[추정]"으로 명시하고, 평론 인용은 매체+점수+링크, 판매/매출 통계는 출처(매체명+연도)를 병기한다.

3. **저장** — `NN_슬러그_연도.md` 형식(예: `38_hades_2020.md`)으로 한두 번의 Write에 저장한다. 조금씩 여러 번 append하지 마라.

분석 깊이에 대한 의사결정이 필요하면 **항상 더 디테일하고 정보량 많은 쪽**을 택한다. 절대 짧은 분석으로 끝내지 마라.

## 마크다운 표기 규칙 (필수)

마크다운 렌더러는 `<단어>`를 HTML 태그로 해석해 렌더링 시 사라지거나 깨뜨린다. 이 때문에 과거 분석서가 대량으로 망가진 적이 있다. 그래서:

- **작품명·게임명·매체명은 꺾쇠 `<>`가 아니라 겹화살괄호 《 》로 표기**한다. 예: `<Elden Ring>` ❌ → `《Elden Ring》` ✅
- **부등호**(`<18kg`, `<10일`)는 "18kg 미만", "10일 이내" 같은 자연어로 쓰거나 `&lt;`로 이스케이프한다.
- 표는 표준 마크다운 표, 강조는 `**볼드**`/`*이탤릭*`, 링크는 `[텍스트](URL)`. 백틱은 코드/명령어에만 쓴다.

## 대량 자동화 모드

게임 수십 개 이상을 멈추지 않고 자동 생성하려면 `references/automation-setup.md`를 읽고 그대로 따른다. 핵심 구조만 요약하면:

- **마스터 상태 파일** `_progress.json` — 게임별 `{id, title, year, slug, filename, status}`.
- **워커** `scripts/worker.ps1` — OS 스케줄러가 N분마다 실행. 매 사이클 **실제 산출물 .md 파일을 스캔해 상태를 재생성**하고(신뢰 가능한 단일 진실), pending을 6개씩 sub-agent로 병렬 분석시키며(`claude -p`), pending이 0이면 새 후보를 자동 발굴해 무한 반복한다.
- **스케줄러 등록** `scripts/setup_scheduler.ps1` — Windows Task Scheduler에 15분 주기로 등록.
- **실시간 모니터** `monitor/start-monitor.ps1` — 워커 진행(상태·완성수·진행률·지금 분석 중·활동 로그)을 브라우저로 보는 대시보드(Node.js, 의존성 0). **대량 자동화를 시작하거나 제어(Enable/Start/Disable)할 때 이 모니터를 함께 띄운다**(멱등 실행 — 이미 떠 있으면 브라우저만 연다). 상세: `references/automation-setup.md`의 "실시간 모니터".

이 구조가 quota(5시간 사용 한도)에 걸려도 멈추지 않는 이유: 워커는 quota가 없으면 그냥 종료하고, 다음 스케줄 사이클에서 다시 시도한다. quota가 리셋되면 자동으로 이어서 진행된다.

## 실패에서 배운 핵심 교훈 (반드시 지킬 것)

이 패턴은 실제 운영에서 두 번 크게 망가졌다가 고쳐졌다. 같은 실수를 반복하지 마라:

1. **sub-agent는 `_progress.json`을 절대 읽거나 쓰지 않는다.** 분석서 .md만 저장한다. 여러 sub-agent가 동시에 상태 파일을 고치면 race condition으로 파일이 오염되고("전부 완료"로 잘못 기록됨), 워커가 오판해 멈춘다. 상태 관리는 워커가 단독으로, 실제 파일 존재 여부로만 판정한다.

2. **동시 실행은 6개 정도로 제한하고 WebFetch를 절제한다.** 한 번에 16~36개를 띄우고 각자 WebFetch를 수십 번 호출하면 5시간 quota를 순식간에 태워 대부분이 미완성으로 중단된다. 적게 띄워 각 건을 확실히 완성시키는 편이 총 산출이 많다.

3. **완성 판정은 결과물로 한다.** 워커는 progress.json의 기존 status를 믿지 말고, 매번 실제 .md 파일이 존재하고 충분한 크기(16KB 이상 권장)인지로 completed를 판정한다. 이러면 어떤 오염도 다음 사이클에 자동 교정된다.

## 번들 리소스

- `references/analysis-template.md` — 8개 섹션의 상세 작성 지침. 단건·대량 모두 이걸 따른다.
- `references/automation-setup.md` — 대량 자동화 전체 셋업 절차(스케줄러 등록, 워커 동작, 제어 명령, 트러블슈팅).
- `scripts/worker.ps1` — 워커 PowerShell 템플릿. 경로/스케줄명만 바꿔 쓴다.
- `scripts/setup_scheduler.ps1` — Task Scheduler 등록/제어 스크립트.
- `assets/progress_template.json` — `_progress.json` 초기 템플릿.

> **실시간 모니터**(`monitor/server.js`·`dashboard.html`·`start-monitor.ps1`)는 스킬 번들이 아니라 **프로젝트 루트의 `monitor/`**에 있다. 워커 진행을 시각화하며 `start-monitor.ps1`로 연다. 대량 자동화 시작/제어 시 함께 띄운다.

> **⚠ PowerShell 스크립트 인코딩**: 한글 리터럴이 든 `.ps1`(워커·런처 등)은 반드시 **UTF-8 with BOM**으로 저장한다. Windows PowerShell 5.1은 BOM 없는 `.ps1`을 CP949로 읽어 한글이 파싱 단계에서 깨지고, 그게 로그·sub-agent 프롬프트로 전파된다. 첫 3바이트가 `EF BB BF`인지 확인하라.
