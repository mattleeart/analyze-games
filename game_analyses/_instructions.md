# 게임 분석 작업 지시서 (서브에이전트용)

## 작업 개요
각 게임에 대해 **장문의 기자 리뷰 수준** 분석서를 작성한다. 영어권 자료(영어 매체, GDC, 공식 인터뷰, Reddit/메가스레드 등) 위주로 매우 자세하게 리서치한다.

## 결과물 형식
- 파일명: `NN_게임슬러그_연도.md` (예: `01_overwatch_2016.md`)
- 저장 위치: `E:/ClaudeProject/remote-work-001/game_analyses/`
- 언어: 한국어 (단, 영어권 원자료 인용은 원문 + 한국어 의역 병기)
- 분량 기준: **최소 8,000자 이상**, 가능하면 15,000자~25,000자 (장문 기자 리뷰 수준)

## ⚠️ 마크다운 문법 준수 (필수)
- **꺾쇠괄호 `<` `>`로 텍스트를 감싸지 말 것.** 마크다운이 HTML 태그로 해석해 해당 부분이 렌더링 시 사라지거나 깨진다. (과거 발생한 버그)
- **작품명·게임명·매체명** 표기: 꺾쇠 대신 한국어 작품명 기호 **《 》**(겹화살괄호) 사용. 예: `<U4>` ❌ → `《U4》` ✅, `<The Last of Us>` ❌ → `《The Last of Us》` ✅
- **부등호**: `<18kg`, `<10일` 같은 표현은 "18kg 미만", "10일 이내" 등 **자연어**로 쓰거나 `&lt;`로 이스케이프.
- 표는 표준 마크다운 표 문법(`| 헤더 |`), 강조는 `**볼드**`/`*이탤릭*`, 링크는 `[텍스트](URL)` 사용.
- 코드/명령어가 아닌 한 백틱(`` ` ``) 남용 금지.

## 필수 분석 항목 (목차 구조)

### 1. 게임 개요
- 개발사/퍼블리셔, 출시일(플랫폼별), 장르
- 디렉터/리드/주요 크레딧
- 개발 기간, 개발 규모(인력/예산)
- 엔진/기술 스택

### 2. 게임 설명 (이 게임이 뭔지 디테일하게)
- 컨셉, 세계관, 줄거리(스포일러 포함, 단 [스포일러] 마크)
- 캐릭터/주요 NPC
- 무드/톤/아트 디렉션
- 사운드/음악 (작곡가, 명곡 사례)

### 3. 핵심 시스템 / 메카닉 (가장 자세하게)
- 코어 게임플레이 루프 (모먼트-투-모먼트)
- 진행 구조 (챕터/맵/오픈월드/로비/메타)
- 전투/주요 메카닉의 정밀한 동작 설명
- 자원 관리, 경제, 진척도 시스템
- 멀티/협력 시스템 (해당 시)
- 라이브 운영/시즌 패스/MTX (해당 시)
- 난이도/접근성 옵션
- UI/UX 디자인 철학

### 4. 평가
- Metacritic 평론/유저 스코어, 주요 평론 인용 (IGN, GameSpot, Eurogamer, PC Gamer, Polygon, Edge, Kotaku, RPS, Destructoid, OpenCritic 등)
- 수상 이력 (TGA, BAFTA, GDCA, DICE, IGF, GOTY 등)
- 상업 지표 (판매량, 동접, 매출, 플레이어 수)
- 유저 평가 (Steam, Reddit, 메가스레드)
- 평론-유저 괴리 (있다면)

### 5. 성공 요인 분석 (핵심)
- 디자인 측면 성공 요인
- 마케팅/출시 전략
- 타이밍/시장 환경
- 개발/운영 방법론
- 커뮤니티/IP 효과
- 비교 가능한 동시기 작품 대비 차별점

### 6. 비평적 시각 / 한계 / 논란
- 디자인적 약점
- 운영/논란 이슈
- 평가가 갈리는 지점

### 7. 영향과 유산
- 장르에 미친 영향
- 후속작/모방작/장르 확장
- 산업적 의미
- 문화적 의미

### 8. 부록
- GDC 강연/포스트모템 자료 링크
- 주요 인터뷰
- 핵심 통계 표
- 참고 자료 목록 (영어권 매체 중심)

## 작업 지침
1. **영어권 자료 우선**: IGN, GameSpot, Polygon, Eurogamer, PC Gamer, RPS, Edge, Kotaku, Game Informer, Variety, Bloomberg, GamesIndustry.biz, GDC Vault, Famitsu(영문판) 등
2. **WebSearch 위주로 리서치 (quota 효율 중요)**:
   - **WebSearch를 주력으로 사용** (8~12회). WebSearch 결과 요약만으로도 대부분의 정보 확보 가능.
   - **WebFetch는 꼭 필요한 핵심 출처 3~5개로 제한**. WebFetch는 전체 페이지를 가져와 토큰을 크게 소모하므로 남용 금지. GDC 포스트모템, 공식 개발자 인터뷰, 핵심 판매 데이터 등 깊이 필요한 경우에만.
   - 같은 내용을 반복 검색하지 말 것. 검색 한 번에 충분한 정보를 얻도록 키워드를 잘 구성.
3. **추측 금지**: 확인된 정보만 인용. 추측은 "[추정]" 명시
4. **숫자/날짜 정확성**: 모든 판매량/매출은 출처 명시 (예: "Newzoo 2024 보고서에 따르면")
5. **인용 시 출처 명시**: 평론 인용은 매체+점수+링크
6. **추측되는 두 가지 옵션이 있을 때 더 디테일하고 정보량 많은 쪽 선택**

## ⚠️ 중요: progress.json을 절대 수정하지 말 것
- **`_progress.json` 파일을 읽거나 쓰지 마라.** 진행 상태는 워커가 실제 .md 파일 존재 여부로 자동 판정한다.
- 여러 sub-agent가 동시에 progress.json을 수정하면 race condition으로 파일이 오염되어 전체 작업이 멈춘다(과거 발생한 버그).
- 너의 임무는 **담당 .md 분석서**와 **`_wip/` 리서치 노트**를 저장하는 것뿐이다. progress.json은 절대 손대지 않는다.

## 효율적 작업 순서 (quota 절약 + 중단 대비 체크포인트)
워커가 프롬프트로 **저장 경로**와 **리서치 노트 경로(`_wip/NN_slug.notes.md`)**를 함께 준다. 반드시 이 순서를 지켜라:
1. `_instructions.md` Read (이 파일)
2. **노트 재사용**: 리서치 노트 경로에 파일이 이미 있으면 Read로 읽어라. 거기 정리된 자료를 최대한 재사용하고 부족한 부분만 가볍게 보강하라(WebFetch 생략/최소화). → 이전에 quota로 중단된 작업을 재조사 없이 이어받는 길이다.
3. 노트가 없으면 WebSearch 8~12회(+꼭 필요한 핵심 출처만 WebFetch 3~5회)로 리서치
4. **체크포인트 저장(본문보다 먼저)**: 리서치를 마치는 즉시 핵심 사실·수치·인용·출처(매체+점수+링크)를 리서치 노트 경로에 Write로 저장하라. 작성 도중 quota가 끊겨도 이 노트가 남아 다음 사이클이 재조사를 건너뛴다.
5. 분석서를 한 번에 작성하여 저장 경로에 Write (한 게임 = 본문 Write 1~2회. 조금씩 여러 번 append 금지)
6. 종료. progress.json 건드리지 말 것. (완성되면 워커가 노트를 자동 정리한다)

## 실패 시 대처
- WebSearch 실패 → 다른 키워드/매체로 우회
- 정보 부족 → 확보한 정보로 최대한 충실히. 절대 빈/짧은 분석서로 끝내지 말 것 (최소 8000자)
